## When to Use This Skill

Use when generating or editing permission set metadata, or when granting object, user, and app permissions.

## Framework policy: object-level access only — NO per-field FLS

This framework grants access at the **object level** using `viewAllFields=true` and `editAllFields=true` on `<objectPermissions>`. It does **NOT** generate `<fieldPermissions>` blocks.

**Why this rule exists.** Per-field FLS in `PermissionSet` XML is the single largest cause of deploy failures in this framework: required fields, formula fields, master-detail relationships, auto-numbers, compound fields, and platform-managed system fields are all FLS-ineligible and reject deploy. Maintaining a clean per-field allowlist across schema drift is brittle and consistently fails. Object-level `viewAllFields`/`editAllFields` is correct, deployable, and survives schema changes.

**Hard rule.** Permission sets generated by this framework MUST NOT contain any `<fieldPermissions>` block. The pre-flight script blocks deploy when one is present.

If you need to restrict access to a sensitive field, do it by:
1. Putting the object in a **separate permission set** that only the right persona receives, OR
2. Using **record-level sharing** (sharing rules, criteria-based sharing, manual share), OR
3. Encrypting the field (Shield Platform Encryption) and gating decryption with a separate permission.

Never address it with `<fieldPermissions>` in the permission set.

## Step 1: Define Core Properties

```xml
<PermissionSet xmlns="http://soap.sforce.com/2006/04/metadata">
    <fullName>YourPermissionSetName</fullName>
    <label>Display Name for Administrators</label>
    <description>Clear description of purpose and intended audience</description>
    <license>Salesforce</license>
    <hasActivationRequired>false</hasActivationRequired>
</PermissionSet>
```

**Naming conventions:**
- Format: `[AppPrefix]_[Component]_[AccessLevel]` (AppPrefix from `.adlc/config.yml` `salesforce.app_prefix`)
- `AccessLevel` ∈ `Read`, `Write`, `Full`, `Execute`, `Admin`
- Examples: `SalesApp_Opportunity_Read`, `OrderMgmt_Product_Write`

## Step 2: Configure Object Permissions (object-level access)

This is the canonical block this framework emits for any object the persona needs access to. Set the CRUD bits to what the persona needs and ALWAYS set both `viewAllFields=true` and `editAllFields=true` so all eligible fields on the object are visible/editable without per-field FLS:

```xml
<objectPermissions>
    <allowCreate>true</allowCreate>
    <allowRead>true</allowRead>
    <allowEdit>true</allowEdit>
    <allowDelete>false</allowDelete>
    <viewAllFields>true</viewAllFields>
    <editAllFields>true</editAllFields>
    <viewAllRecords>false</viewAllRecords>
    <modifyAllRecords>false</modifyAllRecords>
    <object>Account</object>
</objectPermissions>
```

Notes:
- `editAllFields=true` requires `allowEdit=true`. For a Read-only set, use `viewAllFields=true` and leave `editAllFields=false`.
- `viewAllFields` / `editAllFields` are **field-level** flags scoped to the object — they do NOT grant `View All Records` / `Modify All Records`. Record visibility still respects sharing.
- Keep ≤10 different objects per permission set; split into focused sets when more are needed.
- Never combine Read + Delete on the same object in a single set — split into two sets.

### Read-only set example

```xml
<objectPermissions>
    <allowCreate>false</allowCreate>
    <allowRead>true</allowRead>
    <allowEdit>false</allowEdit>
    <allowDelete>false</allowDelete>
    <viewAllFields>true</viewAllFields>
    <editAllFields>false</editAllFields>
    <viewAllRecords>false</viewAllRecords>
    <modifyAllRecords>false</modifyAllRecords>
    <object>Order__c</object>
</objectPermissions>
```

## Step 3: DO NOT emit `<fieldPermissions>`

Per the framework policy above, do not generate any `<fieldPermissions>` block. The decision tree is short:

1. Persona needs access to the field? → Make sure the object is in this set with `viewAllFields=true` (and `editAllFields=true` if write is needed). Done.
2. Persona must NOT see the field? → Do not put the object in this set; route them through a different set, or restrict via sharing / encryption.

The pre-flight script (`tools/sf-preflight/check.sh permsets`) BLOCKS any permission set that contains `<fieldPermissions>`. This is a hard gate, not a warning.

## Step 4: Grant User Permissions

Add system-level permissions for features and capabilities:

```xml
<userPermissions>
    <enabled>true</enabled>
    <name>ApiEnabled</name>
</userPermissions>
<userPermissions>
    <enabled>true</enabled>
    <name>RunReports</name>
</userPermissions>
```

**Common permissions:** `ApiEnabled`, `ViewSetup`, `RunReports`.

**NEVER grant in functional permission sets:** `ViewAllData`, `ModifyAllData`, `ManageUsers`. These are reserved for admin-only sets reviewed by security.

## Step 5: Configure App and Tab Visibility

```xml
<applicationVisibilities>
    <application>Sales_Console</application>
    <visible>true</visible>
</applicationVisibilities>
<tabSettings>
    <tab>CustomTab__c</tab>
    <visibility>Visible</visibility>
</tabSettings>
```

**Tab visibility options:**
- `Visible` — appears on All Tabs and in the visible tabs for its app
- `Available` — available on All Tabs page; users opt in
- `None` — not visible

**Tab naming:**
- Custom object tabs: `__c` suffix (e.g., `MyCustomObject__c`)
- Standard object tabs: `standard-` prefix (e.g., `standard-Account`)

## Step 6: Add Apex and Visualforce Access (Optional)

```xml
<classAccesses>
    <apexClass>CustomController</apexClass>
    <enabled>true</enabled>
</classAccesses>
<pageAccesses>
    <apexPage>CustomPage</apexPage>
    <enabled>true</enabled>
</pageAccesses>
```

## Step 7: Record Type Visibilities (Optional)

```xml
<recordTypeVisibilities>
    <recordType>Account.Business</recordType>
    <visible>true</visible>
    <default>true</default>
</recordTypeVisibilities>
```

## Step 8: Set Agent Access (Optional)

Enable access to Agentforce Employee Agents for users assigned to this permission set:

```xml
<agentAccesses>
    <agentName>Sales_Assistant_Agent</agentName>
    <enabled>true</enabled>
</agentAccesses>
```

- `agentName` (Required): developer name of the employee agent
- `enabled` (Required): `true` to grant access, `false` to deny

## Validation Checklist

Before deploying, verify:
- [ ] `fullName`, `label`, `description` set
- [ ] **No `<fieldPermissions>` blocks anywhere in the file** (framework policy)
- [ ] Every `<objectPermissions>` block has `viewAllFields=true` (and `editAllFields=true` when `allowEdit=true`)
- [ ] No `ViewAllData` / `ModifyAllData` in `<userPermissions>`
- [ ] ≤10 objects in `<objectPermissions>`
- [ ] No combined Read + Delete on the same object — split if so
- [ ] Naming follows `[AppPrefix]_[Component]_[AccessLevel]`
- [ ] No duplicate permissions

## Local pre-flight (always run before deploy)

```sh
sh tools/sf-preflight/check.sh permsets \
  --workspace force-app \
  --target-org "$ALIAS"
```

The script BLOCKS when:
- Any `<fieldPermissions>` block is present (framework policy violation)
- Any `<objectPermissions>` block is missing `viewAllFields=true` (or `editAllFields=true` when `allowEdit=true`)
- `ViewAllData` / `ModifyAllData` is granted in `<userPermissions>`
- Any object name doesn't exist in the target org (when not `--offline`)

Add `--offline` to skip the org describe (use during pure authoring; never as the final gate). Add `--json` for machine-readable output. Exit codes: `0` = clean, `1` = at least one finding (BLOCK), `2` = invocation error.

## What Causes Deployment Failure (and how this policy avoids it)

- **Per-field FLS on required / formula / master-detail / auto-number / compound / system fields** — the historical #1 deploy failure. **This framework eliminates the entire class** by not emitting `<fieldPermissions>` at all.
- **Field doesn't exist in target org** — same root cause; eliminated for the same reason.
- **Object doesn't exist in target org** — caught by pre-flight (`--target-org` describe).
- **Incorrect API names** — wrong tab prefix, missing `__c`, namespace mismatch — caught by pre-flight and at deploy time.

## Deployment

Deploy using Salesforce CLI:

```sh
sf project deploy start --source-dir force-app/main/default/permissionsets --target-org "$ALIAS"
```
